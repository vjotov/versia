Hints from Other Users

Here are suggestions from other Sun WorkShopTM TeamWare tools users. Check with the administrator of your site for guidelines specific to your site.

  • Do bringovers often. Changes may occur in the parent workspace that affect your work.

  • Click the Entire Workspace button to bring over or put back all files in a workspace, or click the Add button to select specific files.

  • To select only a few files from a long list, click Select All, then click the files you want to keep to deselect them, then click Delete. Only the files you clicked remain.

  • If you are using Sun WorkShop TeamWare configuration management tools with the Sun WorkShopTM development environment for source code, execute a build in the Building window after a bringover. This will propagate your changes throughout your application.

  • Split large workspaces into smaller workspaces. Divide a workspace into reasonable subgroups, and create workspaces from those.

  • Subdivide workspace directories for better performance. If you have over 2,000 files in a single directory, you could experience long delays when executing bringovers and putbacks.

  • Remember that when you create a freezepoint, only the differences between the current files and the original hierarchy is recorded. You must have the original workspace (or its parent or child) to extract from a freezepoint file.

  • Keep your .cshrc file small if you plan to use the distributed make feature. Each time dmake accesses a new machine, it reads .cshrc, so a large .cshrc file slows the process.

  • Avoid using the sccs subcommands cdc, comb, fix, and rmdel in TeamWare workspaces. Using these commands can alter SCCS history files in ways which will make them impossible for TeamWare to track. These commands also have undesirable side effects when used on files that exist in multiple workspaces that eventually may be merged.
    cdc Can cause unnecessary branching and confusing histories.
    comb Completely rebuilds the SCCS history file.
    fix A front-end for rmdel.
    rmdel SCCS restricts the use of the rmdel command to remove only the most recent (leaf) delta on a branch. If you remove a delta that also exists in another workspace, it is possible that another user will add a delta in the other workspace. The delta that was removed in your workspace will no longer be a leaf delta when the files are merged.
